## 1. Контекст проекта

**VibeFly** — phone-as-a-server PaaS. Превращает Android-телефон (root + Magisk/KernelSU) в self-hosted сервер для деплоя приложений с GitHub-репозиториев.

- **Репо:** `igor1000rr/vibefly`
- **Стек агента:** Go 1.22, chi router, websocket nhooyr, pure Go (CGO_ENABLED=0), arm64 target
- **Стек Android:** Kotlin + Jetpack Compose, embedded foreground service, EncryptedSharedPreferences для токенов
- **CI:** GitHub Actions, собирает arm64 агент → bundling в Android APK → upload artifact
- **Тест-устройство:** рутованный arm64-смартфон (Magisk)

**Целевой пользователь** — разработчик, который хочет хостить свои pet-проекты, Telegram-боты, веб-сервисы на старом телефоне дома без VPS.

### 3.1. auth_token auto-generation

- RuntimeManager при первом старте генерит `SecureRandom` 32 байта → hex string (64 символа)
- Сохраняется в SettingsStore (EncryptedSharedPreferences)
- Пишется в `agent.toml` как `auth_token`
- Все API кроме `/health` требуют `Authorization: Bearer <token>`
- ServiceLocator реактивно пересоздаёт AgentClient при изменении токена
- `seed_demo_apps = false` в реальном APK (раньше было true → 4 фейк-приложения в проде)

**Закрывает дыру:** до этого при включённом туннеле API был открыт всему интернету.

### 3.2. Real device metrics через root

Новый пакет `apps/agent/internal/rootx/`:

- `Available() bool` — есть ли su (Magisk проверяет `/system/bin/su` + `id` возвращает `uid=0`)
- `Run(ctx, shellCmd)` — выполнить через `su -c`
- `ReadFile(ctx, path)` — сначала `os.ReadFile`, fallback на `cat` через su (для путей заблокированных SELinux)
- `WriteFile(ctx, path, content)` — `echo > path` через su (для cgroup writes)

`metrics/metrics.go` теперь использует rootx для battery (`/sys/class/power_supply/*/capacity`) и thermal (`/sys/class/thermal/thermal_zone*/temp`). Без root — fallback на что прочитается без root (CPU loadavg, /proc/meminfo).

Snapshot включает поле `RootAvailable bool` для UI.

### 3.3. Cgroup v2 limits

`apps/agent/internal/supervisor/cgroup.go`:

- `applyCgroupLimits(logger, pid, spec)` — создаёт `/sys/fs/cgroup/vibefly-<id>`, пишет `memory.max` и `cpu.max`, перемещает pid в `cgroup.procs`
- Все дочерние процессы автоматически наследуют лимиты
- Без root тихо игнорируется (limits opt-in, не критично)
- `cleanupCgroup` при Uninstall (rmdir пустого cgroup)
- `parseMemoryMax`: "512M" → "536870912", "1G" → "1073741824", "0"/"max" → "max"
- `parseCPUQuota`: "50%" → "50000 100000", "200%" → "200000 100000" (2 ядра)

Вызывается из `ExecSupervisor.startWithSpec` после `cmd.Start()`.

### 3.4. Deploy from GitHub repo URL

Backend готов целиком (UI ещё нет — можно тестить curl'ом).

`apps/agent/internal/manifest/manifest.go`:

- `ParseGitHubURL(raw)` → (owner, repo) из `https://github.com/owner/repo` или `git@github.com:owner/repo.git`
- `FetchFromGitHub(ctx, owner, repo, branches)` → скачивает `raw.githubusercontent.com/<owner>/<repo>/<branch>/vibefly.toml` (пробует main → master), 64KB max, parsing через `pelletier/go-toml/v2`

`apps/agent/internal/binstore/archive.go`:

- `DetectArchive(urlOrPath)` → "tar.gz"/"zip"/"" по расширению
- `ExtractArchive(archivePath, destDir, kind)` — pure Go (`archive/tar` + `archive/zip` + `compress/gzip`)
- `safeJoin` защищает от path traversal (`..` в именах внутри архива)
- LimitReader 200MB на файл (защита от zip-bomb)
- Symlinks/devices/fifo игнорируются
- Возвращает путь к первому executable-файлу (mode & 0o111)

`binstore/downloader.go`:

- `DownloadAndExtract(ctx, archiveURL, workdir)` — download + extract + cleanup
- `DownloadBinary(ctx, binaryURL, workdir)` остался для одиночного бинаря в `workdir/binary`

`server/router.go`:

- `POST /apps/from-repo` `{repo_url, branch?, id?}` — главный новый endpoint
- `GET /manifest/preview?repo_url=...&branch=...` — превью без install для UI

**Формат `vibefly.toml`** (см. `docs/repo-deploy.md`):

```toml
name = "My App"
binary_url = "https://github.com/.../releases/download/v1/myapp-arm64.tar.gz"
start_cmd = "./myapp serve"
port = 8080
memory_max = "256M"
cpu_quota = "50%"
autostart = true
restart_policy = "on-failure"

[env]
DATABASE_URL = "sqlite:///data.db"
```

### 3.5. Phase 2.1 — Alpine rootfs bundling

`.github/workflows/ci.yml`:

- Job `fetch-alpine-rootfs` качает `alpine-minirootfs-3.20.3-aarch64.tar.gz` (~3MB), sanity-check на busybox внутри, upload artifact
- `build-android` needs += [..., fetch-alpine-rootfs]
- Download step кладёт тарбол в `apps/android/app/src/main/assets/rootfs/`

`apps/agent/internal/rootfs/manager.go`:

- `Manager{baseDir, tarballPath, logger}`
- `EnsureBase(ctx)` идемпотентно распаковывает tarball → baseDir:
  - sentinel-файл `baseDir/.vibefly-rootfs-ready` (skip если есть)
  - host `tar` через `exec.Command` first, fallback на `su -c "tar..."` если EACCES (для setuid файлов в Alpine)
  - `removeAllBestEffort` для cleanup полу-состояния (если предыдущий extract упал)
- `IsReady() bool` — проверка sentinel

`apps/agent/internal/config/config.go`:

- + `RootfsBaseDir`, `RootfsTarballPath`
- Если оба пусты → chroot не инициализируется (обратная совместимость с phase 1)

`apps/agent/cmd/agent/main.go`:

- `Version = "0.2.0-dev"`
- `rootfs.NewManager` + `EnsureBase()` в **фоновой горутине** (не блокирует HTTP-старт, ~5-10 сек на телефоне)

`apps/android/.../runtime/RuntimeManager.kt`:

- `extractFromAssets("rootfs/alpine-minirootfs.tar.gz")` → `filesDir/rootfs/alpine-minirootfs.tar.gz`
- `buildConfig` принимает `rootfsTarball`+`rootfsBaseDir`, пишет в `agent.toml`
- `Status.rootfsBundled: Boolean` поле

## 5. Ingress / tunneling — обоснование решения

> На момент написания это был открытый блокирующий вопрос. Сейчас закрыт: ingress = self-hosted Pangolin (см. `ROADMAP.md`, этап 2). Ниже — почему.

### Проблема

Телефон сидит за CGNAT оператора или за домашним роутером — публичного IP нет, порты не прокинуть. Reverse-tunnel наружу обязателен, иначе хостить можно только в своей же Wi-Fi-сети. Ingress — обязательный компонент продукта, не опция.

### Cloudflare нам не подходит

Текущая реализация в репо — Cloudflare Tunnel (`cloudflared` бандлится в APK ~27 MB; `tunnel.Cloudflared` + `tunnel.AppTunnels`). Работает, но есть требование: ingress не должен зависеть от Cloudflare. Поэтому cloudflared выпиливается (этап 2.1).

### Что выбрали: self-hosted Pangolin

**Pangolin** (`fosrl/pangolin`) — open-source self-hosted reverse-proxy + WireGuard. Клиент **Newt** (Go-бинарь, user-space WireGuard, root не требует). Ставится на собственный VPS рядом с уже имеющимся Traefik и Let's Encrypt — без новых расходов и без vendor lock-in.

Лицензионная гигиена: Pangolin/Newt под AGPL-3 — в APK не бандлим, Newt качаем при первом старте из официальных releases и запускаем отдельным процессом.

Один туннель закрывает два юзкейса: admin-dashboard и публичные URL приложений на поддоменах `*.vibefly.app`.

### Рассмотренные альтернативы

Сравнивались Cloudflare Tunnel (отклонён — зависимость от CF), ngrok (платный для серьёзного использования, lock-in), Tailscale/Headscale (mesh-P2P — оставлен как возможный параллельный backend для доступа в локалку без публикации), Pangolin Cloud free tier (lock-in на pangolin.net). Победил self-hosted Pangolin: полный контроль, ноль новых расходов, инфраструктура (VPS + Traefik + сертификаты) уже есть.

Отдельный нюанс: если площадка размещения раздаёт только IPv6, а большинство клиентских провайдеров — IPv4-only, нужен relay через узел с публичным IPv4. Это учитывается при выборе, где поднимать Pangolin.

### Droidspaces — рассмотрен, не интегрируется

**Droidspaces-OSS** (`ravindu644/Droidspaces-OSS`) — низкоуровневый контейнер-рантайм для Android (namespaces, systemd, cgroups). Требует custom kernel (CONFIG_NAMESPACES/CGROUPS/SECCOMP) — большинство стоковых ядер не подходит. GPL-3 — бандлинг залочил бы весь VibeFly под GPL. Решение: не интегрируем; остаётся опциональным exec-backend в будущем (см. `ROADMAP.md`, этап 3, ветка A).

### План интеграции Pangolin (этап 2)

1. **Удалить cloudflared целиком** — CI job, Android assets, `tunnel/cloudflared.go`, `tunnel/app_tunnels.go`, UI Publish/PublicUrlCard. Чистый «минусовый» коммит.
2. **Добавить Newt-клиент** — `tunnel.PangolinManager` реализует `tunnel.Manager`. Download Newt при первом старте (не бандлим), запуск через `su -c`, reconnect-loop. Конфиг: `pangolin_endpoint`, `newt_id`, `newt_secret` в `agent.toml`.
3. **Android UI** — Settings → Pangolin: три поля + кнопка «Подключиться» + индикатор состояния.

## 6. Технический контекст

### Стиль работы

- **Commits / комментарии / docs — на русском.** Идентификаторы (функции, переменные, файлы) — на английском.
- **Push прямо в main через gitv2 MCP.** Без PR.
- **GitHub-токены в чат не вставляются и из контейнера не пушатся.** gitv2 MCP уже авторизован.
- **Атомарные коммиты** — каждый самодостаточен, CI зелёный, безопасен к откату.
- **Не плодить лишние md-файлы.** Только если реально нужно.

### Среда исполнения

- vibefly использует **свою CI на GitHub Actions** (не Coolify). Сборка в Docker-контейнере, артефакты — APK + arm64-бинарь агента.

### Android-target

- Минимальный API level 26 (Android 8.0), целевой 34.
- arm64 only — armeabi-v7a и x86_64 не поддерживаем.
- Foreground service обязателен, иначе Android убьёт агента в фоне (LMK).
- Magisk root ожидаем — без него часть фич не работает (cgroups, реальные метрики через SELinux-блокированные пути).
- SELinux Enforcing даже с root блокирует часть путей → отсюда rootx fallback через `su -c "cat ..."`.

## 7. Что делать в следующей сессии

Решение по ingress принято (Pangolin) — идём по `ROADMAP.md`, этап 2 (выпил cloudflared → PangolinManager → Android UI).

Если сначала тестируем железо — чек-лист последнего APK на реальном телефоне:

1. Установить APK с GitHub Actions artifact.
2. `adb logcat | grep VibeFlyRuntime` — при первом старте «сгенерирован новый auth_token (32 байта)».
3. В логах агента: «rootfs tarball извлечён» → «распаковка Alpine rootfs» → «rootfs ready».
4. `adb shell` → `su` → `ls .../files/rootfs-base/` — структура Alpine (bin/ usr/ etc/ ...).
5. `cat .../files/rootfs-base/etc/alpine-release` → `3.20.3`.
6. Settings → metrics: battery + thermal не нулевые (root через Magisk popup).
7. `POST /apps/from-repo` с репозиторием, у которого в Releases есть arm64-тарбол (например Caddy) — деплой end-to-end.
