# VibeFly — Session Handoff (14 мая 2026)

Документ для следующей сессии Claude. Описывает где сейчас проект, что было сделано в последних восьми коммитах, какие архитектурные решения зафиксированы и где открытый блокирующий вопрос.

## 1. Контекст проекта

**VibeFly** — phone-as-a-server PaaS. Превращает Android-телефон (root + Magisk/KernelSU) в self-hosted сервер для деплоя приложений с GitHub-репозиториев.

- **Репо:** `igor1000rr/vibefly` (private)
- **Стек агента:** Go 1.22, chi router, websocket nhooyr, pure Go (CGO_ENABLED=0), arm64 target
- **Стек Android:** Kotlin + Jetpack Compose, embedded foreground service, EncryptedSharedPreferences для токенов
- **CI:** GitHub Actions, собирает arm64 агент → bundling в Android APK → upload artifact
- **Test device:** Redmi Note (точная модель в memory) с Magisk + arm64

**Целевой пользователь** — разработчик, который хочет хостить свои pet-проекты, Telegram-боты, веб-сервисы на старом телефоне дома без VPS.

### 3.1. auth_token auto-generation

- RuntimeManager при первом старте генерит `SecureRandom` 32 байта → hex string (64 символа)
- Сохраняется в SettingsStore (EncryptedSharedPreferences)
- Пишется в `agent.toml` как `auth_token`
- Все API кроме `/health` требуют `Authorization: Bearer <token>`
- ServiceLocator реактивно пересоздаёт AgentClient при изменении токена
- `seed_demo_apps = false` в реальном APK (раньше было true → 4 фейк-приложения в проде)

**Закрывает дыру:** до этого если включал Cloudflare Tunnel — API был открыт всему интернету.

### 3.2. Real device metrics через root

Новый пакет `apps/agent/internal/rootx/`:
- `Available() bool` — есть ли su (Magisk проверяет `/system/bin/su` + `id` возвращает `uid=0`)
- `Run(ctx, shellCmd)` — выполнить через `su -c`
- `ReadFile(ctx, path)` — сначала `os.ReadFile`, fallback на `cat` через su (для путей заблокированных SELinux)
- `WriteFile(ctx, path, content)` — `echo > path` через su (для cgroup writes)

`metrics/metrics.go` теперь использует rootx для battery (`/sys/class/power_supply/*/capacity`) и thermal (`/sys/class/thermal/thermal_zone*/temp`). Без root — fallback на что прочитается без root (CPU loadavg, /proc/meminfo).

Snapshot включает поле `RootAvailable bool` для UI.

### 3.3. Cgroup v2 limits

`apps/agent/internal/supervisor/cgroup.go`:
- `applyCgroupLimits(logger, pid, spec)` — создаёт `/sys/fs/cgroup/vibefly-<id>`, пишет `memory.max` и `cpu.max`, перемещает pid в `cgroup.procs`
- Все дочерние процессы автоматически наследуют лимиты
- Без root тихо игнорируется (limits opt-in, не критично)
- `cleanupCgroup` при Uninstall (rmdir пустого cgroup)
- `parseMemoryMax`: "512M" → "536870912", "1G" → "1073741824", "0"/"max" → "max"
- `parseCPUQuota`: "50%" → "50000 100000", "200%" → "200000 100000" (2 ядра)

Вызывается из `ExecSupervisor.startWithSpec` после `cmd.Start()`.

### 3.4. Deploy from GitHub repo URL

Backend готов целиком (UI ещё нет — можно тестить curl'ом).

`apps/agent/internal/manifest/manifest.go`:
- `ParseGitHubURL(raw)` → (owner, repo) из `https://github.com/owner/repo` или `git@github.com:owner/repo.git`
- `FetchFromGitHub(ctx, owner, repo, branches)` → скачивает `raw.githubusercontent.com/<owner>/<repo>/<branch>/vibefly.toml` (пробует main → master), 64KB max, parsing через `pelletier/go-toml/v2`

`apps/agent/internal/binstore/archive.go`:
- `DetectArchive(urlOrPath)` → "tar.gz"/"zip"/"" по расширению
- `ExtractArchive(archivePath, destDir, kind)` — pure Go (`archive/tar` + `archive/zip` + `compress/gzip`)
- `safeJoin` защищает от path traversal (`..` в именах внутри архива)
- LimitReader 200MB на файл (защита от zip-bomb)
- Symlinks/devices/fifo игнорируются
- Возвращает путь к первому executable-файлу (mode & 0o111)

`binstore/downloader.go`:
- `DownloadAndExtract(ctx, archiveURL, workdir)` — download + extract + cleanup
- `DownloadBinary(ctx, binaryURL, workdir)` остался для одиночного бинаря в `workdir/binary`

`server/router.go`:
- `POST /apps/from-repo` `{repo_url, branch?, id?}` — главный новый endpoint
- `GET /manifest/preview?repo_url=...&branch=...` — превью без installа для UI

**Формат `vibefly.toml`** (см. `docs/repo-deploy.md` в репо):
```toml
name = "My App"
binary_url = "https://github.com/.../releases/download/v1/myapp-arm64.tar.gz"
start_cmd = "./myapp serve"
port = 8080
memory_max = "256M"
cpu_quota = "50%"
autostart = true
restart_policy = "on-failure"

[env]
DATABASE_URL = "sqlite:///data.db"
```

### 3.5. Phase 2.1 — Alpine rootfs bundling

`.github/workflows/ci.yml`:
- Новый job `fetch-alpine-rootfs` качает `alpine-minirootfs-3.20.3-aarch64.tar.gz` (~3MB) с `dl-cdn.alpinelinux.org`, sanity-check на busybox внутри, upload artifact
- `build-android` needs += [..., fetch-alpine-rootfs]
- Download step кладёт тарбол в `apps/android/app/src/main/assets/rootfs/`

`apps/agent/internal/rootfs/manager.go`:
- `Manager{baseDir, tarballPath, logger}`
- `EnsureBase(ctx)` идемпотентно распаковывает tarball → baseDir:
  - sentinel-файл `baseDir/.vibefly-rootfs-ready` (skip если есть)
  - host `tar` через `exec.Command` first, fallback на `su -c "tar..."` если EACCES (для setuid файлов в Alpine)
  - `removeAllBestEffort` для cleanup полу-состояния (если предыдущий extract упал)
- `IsReady() bool` — проверка sentinel

`apps/agent/internal/config/config.go`:
- + `RootfsBaseDir`, `RootfsTarballPath`
- Если оба пусты → chroot не инициализируется (обратная совместимость с phase 1)

`apps/agent/cmd/agent/main.go`:
- `Version = "0.2.0-dev"`
- `rootfs.NewManager` + `EnsureBase()` в **фоновой горутине** (не блокирует HTTP-старт, ~5-10 сек на телефоне)
- `_ = rootfsManager` — переменная будет передана в ChrootSupervisor в следующем коммите (см. далее)

`apps/android/.../runtime/RuntimeManager.kt`:
- `extractFromAssets("rootfs/alpine-minirootfs.tar.gz")` → `filesDir/rootfs/alpine-minirootfs.tar.gz`
- `buildConfig` принимает `rootfsTarball`+`rootfsBaseDir`, пишет в `agent.toml`
- `Status.rootfsBundled: Boolean` поле

`docs/phase2-chroot.md`: roadmap 2.1 (done) → 2.2 (ChrootSupervisor TBD) → 2.3 (overlayfs) → 2.4 (apk add nodejs hook) → 2.5 (UI Runtime radio) → 2.6 (marketplace Node/Python templates).

## 5. БЛОКИРУЮЩИЙ ВОПРОС: ingress / tunneling

Это главный открытый архитектурный вопрос. **Прежде чем продолжать кодить — нужно решить.**

### Сейчас в репо
Используется **Cloudflare Tunnel** (`cloudflared` бинарь bundling в APK, ~27MB). `tunnel.Cloudflared` + `tunnel.AppTunnels` (per-app trycloudflare URLs). Работает.

### Игорь явно сказал: Cloudflare нам не подходит, нужно своё

Это критичное требование. На момент окончания сессии решения не приняли.

### Что обсуждалось и почему не подошло

**Pangolin (fosrl/pangolin)** — open-source self-hosted reverse proxy + WireGuard. AGPL-3 / commercial. Использует Newt client (Go бинарь, user-space WireGuard, не требует root). Технически идеален.

**Но** требует VPS с публичным IPv4. У Игоря:
- В школе (где будет жить ноут-сервер) — **только IPv6**. IPv4 школа не даёт.
- Своих VPS несколько: `45.135.234.145` (Coolify), `185.218.0.7` (NIT), `185.42.14.242` (DJ Radio), `92.205.228.90` (azcrm) — **все имеют IPv4**, но заняты production-сервисами Игоря.

**IPv6-only проблема:** 95% домашних провайдеров в РБ/РФ выдают только IPv4 → клиенты не достучатся до IPv6-only сервера.

**Droidspaces-OSS (ravindu644/Droidspaces-OSS)** — обнаружили в этой сессии через Игоря. Низкоуровневый контейнер-рантайм для Android с настоящими namespaces, systemd, cgroups. Альтернатива нашему планируемому chroot.

**Но** требует custom kernel (CONFIG_NAMESPACES, CONFIG_CGROUPS, CONFIG_SECCOMP) — большинство стоковых kernels не подходит. KernelSU recommended, Magisk supported через Daemon Mode. GPL v3 — bundling в наш APK лочит весь VibeFly под GPL. Решение: **не интегрировать**, оставить как опциональный backend в будущем.

## 6. Технический контекст для будущего Claude

### Стиль работы (зафиксировано пользовательскими memory)
- **Commits/comments/docs на русском.** Идентификаторы (функции, переменные, файлы) на английском.
- **Push directly to main через gitv2 MCP.** Без PR.
- **НИКОГДА не принимать GitHub токены в чате.** gitv2 MCP уже авторизован.
- **push_files требует** owner / repo / branch / message — легко забыть, проверять перед вызовом.
- **MCP gitv2 НЕ умеет delete_file** — для удаления переписать файл как stub либо использовать `push_files` с пустым контентом.
- **Атомарные коммиты** — каждый коммит самодостаточен, CI зелёный, можно откатить безопасно.
- **Не плодить лишние md-файлы.** User preference: "Не пиши кучу шпаргалок, txt и md файлов, все четко и по делу". Только если реально нужно (как этот handoff).
- **Не ждать прохождения CI после push в Veloseller** — пользователь сам мониторит GitHub Actions. Для vibefly — проверять через `gitv2:list_workflow_runs` если важно.

### Среда исполнения
- **Coolify 4.1.0** на `45.135.234.145:8000`. GitHub App `vibecoding-coolify` (ID 3001166). Там же self-hosted Appwrite 1.8.1 на `appwrite.vibecoding.by`.
- **vibefly использует** свою CI на GitHub Actions, не Coolify. Build в Docker контейнере, артефакты — APK + arm64 бинарь агента.
- **Coolify VPS грабли:** Docker 28.x не парсит `host.docker.internal:host-gateway` → фикс через `coolify-fix-host-gateway.sh` после каждого апгрейда. Это касается других проектов Игоря, не vibefly напрямую.

### Что важно понимать про Android-target

- **Минимальный API level 26** (Android 8.0), целевой 34
- **arm64 only** — armeabi-v7a не поддерживаем, x86_64 не имеет смысла для phone-as-a-server
- **Foreground service** обязателен иначе Android убьёт агента через несколько минут (LMK)
- **Magisk root** ожидаем — без него часть фич не работает (cgroups, real metrics через SELinux-блокированные пути)
- **SELinux Enforcing** даже с root блокирует некоторые пути → отсюда rootx fallback через `su -c "cat ..."`
